Track 1 · Technical Agentic Enablement

Six code layers. One production-grade agent.

Every agent on this platform is decomposed into six independently versioned, tested, and deployable code layers. Each has a defined format, runtime, and handoff contract to the layer above and below it. No custom scripts. No notebook-to-production gaps. Structured code from day one.

1
Skill Manifests
YAML SAP AI Core · Skill Registry
The atomic unit. Declares what the agent can do, what parameters it accepts, what tools it can invoke, and what guardrails constrain it. Imported directly into SAP AI Core's Skill Registry or Joule runtime.
# mx-resolver.skill.yaml name: mx-resolver version: "1.0.0" trigger: type: event topic: sap/is-u/meterread/exception/created parameters: - name: exception_id type: string required: true - name: tolerance_pct type: number default: 2.5 tools: - id: get_exception_detail type: odata - id: close_workcase type: bapi - id: post_corrected_reading type: bapi guardrails: max_correction_pct: 5.0 require_human_above_pct: 10.0
What ships per agent
One YAML file per agent archetype. Parameterised for client-specific tolerances. Versioned in git, imported to AI Core via CLI.
2
Action Handlers
Node.js / Python BTP Kyma · CAP
When Joule or AI Core invokes a tool from the skill manifest, it calls an action handler. This is where the real BAPI/RFC/BAdI call happens. Joule never calls SAP directly — handlers own that boundary.
// handlers/closeWorkcase.js import { callBapi } from '../lib/sap-rfc'; import { auditLog } from '../lib/audit'; export async function closeWorkcase({ exception_id, correction_value, agent_trace_id }) { const result = await callBapi('ISU_METERREAD_EXCEPTION_CLOSE', { I_EXCEPTION_ID: exception_id, I_CORRECTION_VAL: correction_value, I_AGENT_REF: agent_trace_id, // audit trail I_CLOSE_REASON: 'AGENT_AUTO', }); if (result.RETURN.TYPE === 'E') throw new AgentActionError(result.RETURN.MESSAGE); await auditLog({ action:'CLOSE', ref:result.E_CHANGEDOC_NO }); return { status:'closed', change_doc: result.E_CHANGEDOC_NO }; }
What ships per agent
3–6 handler functions per agent. Shared lib/sap-rfc wrapper reused across all. Input validation, error handling, and audit log write in every handler.
3
Prompt Templates
Jinja2 SAP GenAI Hub
Prompt engineering is a code artefact, not a conversation. Each agent has a versioned system prompt, a context template that injects SAP data, a few-shot block, and guardrail constraints — all in source control.
{# mx-resolver.system.j2 #} You are MX-Resolver, a goal-driven SAP IS-U exception handling agent. ## Rules - Never correct by more than {{ max_correction_pct }}% without human approval - Always write a change doc before closing - If variance > 3 sigma: ESCALATE ## Context Exception: {{ exception.id }} | Type: {{ exception.type }} Current: {{ exception.current_kwh }} kWh 12M Avg: {{ exception.avg_kwh }} kWh History: {{ exception.history | join(', ') }} ## Decide AUTO_CORRECT | ESCALATE | CLOSE_NO_ACTION Return: decision, corrected_value, reasoning
What ships per agent
System prompt + context template + few-shot block. Deployed to GenAI Hub Prompt Management API. Version pinned in skill manifest so prompt drift is traceable.
4
CDS Views & Data Products
ABAP CDS S/4HANA · IS-U Stack
Agents don't read raw SAP tables. CDS views are the semantic data contract between the SAP system and the agent layer — annotated, OData-published, and filtered to what the agent actually needs.
-- I_MeterReadingExceptionContext.cds @AbapCatalog.sqlViewName: 'ZCDS_MREXCCTX' @OData.publish: true define view I_MeterReadingExceptionContext as select from edmx_exception as ex join isu_meterread_hist as mh on mh.anlage = ex.anlage { key ex.exception_id, ex.exception_type, ex.status, mh.current_kwh, mh.prev_kwh, avg(mh.kwh) over ( partition by mh.anlage rows between 5 preceding and current row ) as trailing_6m_avg } where ex.status = 'OPEN'
What ships per agent
1–3 CDS views per agent domain. OData-annotated for BTP consumption. Delivered as a SAP Transport Request per module. Reusable across multiple agents in the same domain.
5
AI Core Pipelines
Dockerfile + pipeline.yaml SAP AI Core
For agents that need a trained model — propensity scoring, anomaly detection, deduction classification — the training and serving pipeline is a code asset with a proper MLOps lifecycle. Not a notebook.
# pipeline.yaml — SAP AI Core name: collections-propensity-train steps: - name: ingest image: ghcr.io/prasad-ibm/agenticsap/ data-ingest:latest command: ["python", "ingest.py"] env: SOURCE_CDS: I_ContractAccountPaymentHistory LOOKBACK_MONTHS: "24" - name: train image: ghcr.io/prasad-ibm/agenticsap/ propensity-train:latest command: ["python", "train.py", "--model=xgboost"] resources: { cpu: "4", memory: "8Gi" }
What ships per agent
Ingest → train → register → serve Docker images. Reusable base images across all ML-dependent agents. Triggered on schedule or data drift detection.
6
Eval Harnesses
Python pytest GitHub Actions CI/CD
The eval golden set isn't a spreadsheet — it's a test suite that runs on every prompt change or handler update. A CI gate blocks deployment if accuracy drops below threshold. No silent regression.
# tests/test_mx_resolver_eval.py import pytest from agents.mx_resolver import MxResolverAgent from evals.datasets import load_golden_set agent = MxResolverAgent(prompt_version="v1.3.2") golden = load_golden_set("mx_resolver_500.jsonl") @pytest.mark.parametrize("case", golden) def test_decision_accuracy(case): result = agent.decide(case["input"]) assert result["decision"] == case["expected"] def test_never_overcorrects(): # delta must be ≤ 5% of prev reading for case in golden: result = agent.decide(case["input"]) if result["decision"] == "AUTO_CORRECT": assert delta_pct(result, case) <= 0.05
What ships per agent
≥500 labelled examples in JSONL. pytest suite covering decision accuracy, guardrail compliance, and latency. CI gate: must pass 80%+ accuracy before any prompt or handler change ships.
Track 2 · Business & Governance Enablement

Eight artefacts every stakeholder needs before saying yes.

The best code in the world doesn't ship without executive sponsorship, architecture sign-off, and audit-ready evidence. These eight artefacts remove every non-technical blocker — from the CIO's ROI question to the CISO's data lineage question. Each is pre-built per agent, not produced after the fact.

🏗
Architecture Diagram
EASACIO
Colour-coded layer diagram showing trigger → process → data → action → output for each agent. Shows exactly where the agent sits in the SAP landscape and which system boundaries it crosses. Built as a printable HTML/CSS artefact — no Visio dependency.
Deliverable format: HTML / PNG / PDF
📊
Data Product Specification
SAEAData
Documents exactly what data the agent reads, from which CDS views and table joins, with field-level sensitivity classification. Feeds the data governance review and helps basis scope the transport request.
Deliverable format: Markdown / Confluence
🔗
SAP Integration Manifest
SABasisSI
Full inventory of every BAPI, BAdI, RFC, OData endpoint, and CDS view the agent touches — with direction (read/write), key parameters, and SAP release dependency. The exact input to the integration design authority review.
Deliverable format: HTML table / YAML
✅
Eval Golden Set & Scorecard
QARiskAudit
500+ labelled production examples with expected decisions, plus a scorecard showing precision, recall, false positive rate, and guardrail compliance. This is what the change advisory board reviews — not a demo.
Deliverable format: JSONL + HTML scorecard
💰
ROI & Business Case
CIOCFOSponsor
Interactive model: cost inputs (FTE rate, volume, exception rate) drive a live payback calculation with 12/24/36-month NPV. Printable board-pack format. All assumptions sourced and adjustable — not a static slide.
Deliverable format: Interactive HTML / PDF
🧪
Sandbox Simulation
POCSRChampion
A no-SAP-connection walkthrough of the agent's decision loop — real data shapes, synthetic values. The process owner sees the agent reason through five realistic examples. Builds trust before QA access is granted.
Deliverable format: Interactive HTML
🎥
Demo Recording
ExecSteeringSI
A scripted 3-minute walk of the agent end-to-end in a QA SAP system: trigger fires, agent reasons, action happens, audit log written. Used in steering committee decks and SI partner enablement sessions.
Deliverable format: MP4 / Loom link
🛡
Governance & Compliance Pack
CISOAuditDPO
RACI matrix, data lineage diagram, audit log schema, model risk classification (MRC score), human-in-the-loop override protocol, and GDPR data subject access request handling. Everything the DPO and internal audit need in one artefact.
Deliverable format: PDF / Confluence
